# Multi-agent orchestration protocol

**Status:** Canonical baseline  
**Role:** Orchestrator + Subagents (internal helpers)  
**Goal:** Increase depth and throughput via parallel reasoning without sacrificing determinism, auditability, or mpcr hygiene.  
**Change control:** Production-stable. Additive clarifications only; breaking changes require an explicit version bump.

---

## I) Core model

### Roles

- **Orchestrator (you):** The single agent that SHALL be responsible for:
  - producing the final deliverable (UACRP report or dispositions)
  - running mpcr _mutations_ (register/update/note/finalize, set-status, applicator notes), except when explicitly delegating `mpcr reviewer ...` operations to Child Reviewers for their own entry
  - making any repository edits (working tree, commits)

- **Subagents:** Parallel analysts that produce **Proof Packets** for the orchestrator to merge.
  - Default posture: **read-only** (Subagents SHALL be read-only unless explicitly granted write access)
  - Subagents are NOT “reviewers” and are NOT “applicators” for mpcr purposes.
- **Child Reviewers (optional; mpcr-integrated):** Parallel agents that act as full mpcr reviewers and produce their own UACRP reports.
  - The orchestrator still synthesizes the overall outcome, but each Child Reviewer SHALL own its own mpcr reviewer entry (status/notes/report).
  - **Lineage rule (mandatory):** the Child Reviewer SHALL set `parent_id` to the orchestrator’s `reviewer_id` on `mpcr reviewer register` so lineage is recorded.
  - **Reviewer id rule (mandatory):** each Child Reviewer SHALL have a distinct `reviewer_id` within the session (either pre-assigned by the orchestrator or generated by `mpcr reviewer register` and then treated as assigned).
  - Prefer Child Reviewers only when you need independent report artifacts; otherwise prefer Subagents + Proof Packets.

### `mpcr` CLI location (this skill)

Within this skill, `mpcr` refers to `<skills-file-root>/scripts/mpcr`.

## II) Capability detection and fallback

Work SHALL run in **MULTI_AGENT** mode when subagents are available; otherwise it SHALL run in **SINGLE_AGENT** mode using the same decomposition.
The orchestrator SHALL determine availability at the start so the correct mode is selected.

In both modes, the orchestrator SHALL remain the single writer of repository edits and the overall final deliverable.
Child Reviewers (if used) SHALL write artifacts only for their own mpcr reviewer entry (status/notes/report).

### Timing (mandatory; early delegation)

IF subagents / parallel workers are available THEN multi-agent execution SHALL begin early.
You SHALL NOT defer delegation until after you have already performed deep sequential ingestion of the entire codebase/diff/reports.

There SHALL exist:

- a short sequential **Plan/Outline** that defines the decomposition (objectives + disjoint scopes), AND
- parallel delegated work for the remaining depth/coverage effort.

You SHALL ask the user how much parallelism they want (no fixed cap). If the user gives a concurrency budget, you SHALL respect it (use waves if needed); otherwise you SHALL continue delegating until the saturation stop condition defined in Section IV is met.

## III) Determinism and write-ownership rules

### Orchestrator write ownership (default)

The orchestrator SHALL be the only actor that:

- edits repository files
- commits / rebases / pushes
- runs `mpcr reviewer register/update/note/finalize` for the orchestrator’s own reviewer entry
- runs `mpcr applicator set-status` or `mpcr applicator note`

### Reviewer write ownership (Child Reviewers only)

IF the orchestrator delegates a child agent as a **Child Reviewer** THEN that child reviewer SHALL:

- use `mpcr` ONLY for `mpcr reviewer register/update/note/finalize`
- run those commands ONLY for its own `reviewer_id` (assigned by the orchestrator OR captured from `mpcr reviewer register --print-env`) and the shared session context

At minimum, the child reviewer SHALL ensure its mpcr reviewer entry is registered and finalized (typically via `mpcr reviewer register` and `mpcr reviewer finalize`).

Identity rule (mandatory):

- The orchestrator SHALL provide its own `reviewer_id` to the child reviewer as `parent_id`.
- Child reviewers SHALL set `parent_id` to the orchestrator’s `reviewer_id` when registering.
- Each Child Reviewer SHALL use a distinct `reviewer_id` within the session.
  - `reviewer_id` values SHALL be generated by `mpcr` (for example via `mpcr reviewer spawn-children`, `mpcr id id8`, or `mpcr reviewer register --print-env`); they SHALL NOT be hand-written.
  - If the child lets `mpcr reviewer register --print-env` generate `MPCR_REVIEWER_ID`, the child SHALL treat that value as its assigned `reviewer_id` for the remainder of its work and SHALL report it back to the orchestrator.
- Child reviewers SHALL NOT run `mpcr applicator ...` commands.
- Child reviewers SHALL NOT edit repository files (patches/suggestions only).

### Child Reviewer session binding (mandatory)

To keep orchestration deterministic (and to avoid a child joining the wrong session), the orchestrator SHALL ensure the following outcomes:

- **Shared session context:** each Child Reviewer uses the exact same `target_ref` string and passes `--session-dir` and `--session-id` explicitly on every `mpcr reviewer ...` command (Child Reviewers SHALL NOT rely on defaults).
- **Lineage recorded:** each Child Reviewer sets `--parent-id <PARENT_REVIEWER_ID8>` on `register`.
- **Distinct identity:** each Child Reviewer uses a distinct `reviewer_id` (pre-assigned OR generated by `register --print-env` and then treated as assigned).

Non-normative example (preferred; mpcr spawns child entries deterministically):

```sh
# Orchestrator (parent):
mpcr reviewer register --target-ref '<REF>' --print-env
# Copy these values from `--print-env`:
# - <PARENT_REVIEWER_ID8> = MPCR_REVIEWER_ID
# - <DIR> = MPCR_SESSION_DIR
# - <SESSION_ID8> = MPCR_SESSION_ID

# Orchestrator (preferred): spawn child reviewer entries deterministically (creates `_session.json` entries + returns ids):
mpcr reviewer spawn-children --target-ref '<REF>' --session-dir <DIR> --session-id <SESSION_ID8> --parent-id <PARENT_REVIEWER_ID8> --count 1 --json
# Copy a child reviewer id from the output:
# - <CHILD_REVIEWER_ID8> = children[0].reviewer_id

# Child Reviewer (joins the exact session + records lineage):
mpcr reviewer register --target-ref '<REF>' --session-dir <DIR> --session-id <SESSION_ID8> --reviewer-id <CHILD_REVIEWER_ID8> --parent-id <PARENT_REVIEWER_ID8> --print-env
```

Note: IF `parent_id` is set on `register` THEN `--print-env` includes `MPCR_PARENT_ID` for deterministic copy/paste reuse.
If an entry already exists with a missing `parent_id`, re-running `register` with `--parent-id` SHALL record it; if `parent_id` conflicts, `mpcr` SHALL fail (do not overwrite lineage).

Non-normative alternative example (child lets `mpcr` generate its reviewer id; the child reports `MPCR_REVIEWER_ID` back to the orchestrator and uses it for all subsequent commands):

```sh
mpcr reviewer register --target-ref '<REF>' --session-dir <DIR> --session-id <SESSION_ID8> --parent-id <PARENT_REVIEWER_ID8> --print-env
```

IF you are delegated as a Child Reviewer and you cannot run `mpcr` successfully THEN you SHALL stop and tell the orchestrator immediately (do not proceed as a Child Reviewer).

WHEN the orchestrator delegates one or more Child Reviewers THEN the orchestration record SHALL include a `handoff` note so the assignment is auditable via `mpcr`. Include the child reviewer id if it is already known; otherwise, record the assignment and add a follow-up `handoff` note after the child registers.

Non-normative example handoff note:

```sh
mpcr reviewer note --session-dir <DIR> --reviewer-id <PARENT_REVIEWER_ID8> --session-id <SESSION_ID8> --note-type handoff --content-json --content '{
  "children": [
    { "reviewer_id": "<CHILD_REVIEWER_ID8>", "assignment": "Red Team", "scope": "<FILES / ANCHORS>" }
  ]
}'
```

After a Child Reviewer registers, the orchestrator SHALL verify lineage and session binding.

Non-normative example command:

```sh
mpcr session reports open --session-dir <DIR> --session-id <SESSION_ID8> --json
```

Confirm the child entry shows the expected `reviewer_id` and `parent_id` (the orchestrator’s reviewer id).

### Subagent restrictions (default)

Subagents SHALL NOT:

- run `mpcr` commands
- change `initiator_status`
- edit repository files
- write to `_session.json` directly

Subagents MAY:

- read code and diffs
- run read-only inspection commands (e.g., `rg`, `git show`, `cat`)
- propose fixes as text diffs / patch suggestions
- propose verification commands and tests

IF the orchestrator explicitly grants “write access” to a subagent THEN the orchestrator SHALL:

- scope it to specific files/anchors
- state the merge strategy (patch-only; no direct edits preferred)
- state how conflicts will be resolved

## IV) Orchestration outcomes (multi-agent)

This section is outcome-focused: it defines WHAT you need to achieve and WHY.
It intentionally avoids prescribing a single fixed decomposition or a fixed number of agents.

### A) Reviewer outcomes (dynamic; no fixed cap)

WHEN performing a review AND subagents are available THEN the orchestrator SHALL obtain a **dynamic** set of Proof Packets (and/or Child Reviewer UACRP reports) with **disjoint responsibilities**.
The orchestrator SHALL NOT impose an arbitrary upper bound; it SHALL continue delegating work (in waves if needed) until:

- Universal Domain coverage is complete, AND
- additional independent analysis stops producing new failure modes / evidence / mitigations (saturation stop condition).

Required outcomes (why: determinism + depth + auditability):

- **Scope model:** there SHALL exist an anchored change inventory, trust boundary map, and contract surface list.
- **Must‑prove theorems:** there SHALL exist anchored must‑prove theorems for every in-scope Universal Domain.
- **Concrete disproof attempts:** there SHALL exist concrete counterexample attempts for those theorems (even when concluding “no issue found”).
- **Verification plan:** there SHALL exist a verification plan that would fail if key risks still exist (tests/commands/artifacts).
- **Residual risk:** there SHALL exist explicit Assumed/Unknown claims with why-they-matter + how-to-verify.
- **Saturation rationale:** there SHALL exist a recorded reason you stopped delegating (why additional agents would not add new failure modes / evidence).

Non-normative examples (do not treat as an exhaustive or mandatory role list):

- **Scope Mapper:** Produces the scope model (inventory + trust boundaries + depth targets) so downstream proofs are focused.
- **Red Team:** Tries to break correctness/security claims via counterexamples, attack paths, and failure scenarios.
- **Systems Auditor:** Assesses concurrency/perf/operability/dependencies/technical debt and identifies verification gaps.

Subagents and Child Reviewers SHALL tailor their proof effort to the specific change; they SHALL NOT default to a fixed canned role list when the change demands a different decomposition.

IF you need independent report artifacts saved via mpcr THEN you MAY delegate one or more agents as **Child Reviewers** instead of Subagents (see Section III).

### B) Applicator outcomes (provisional; stable baseline)

WHEN applying review feedback AND multiple review reports exist THEN the orchestrator SHALL use a **dynamic** amount of subagent help when available.
The orchestrator SHALL NOT impose an arbitrary upper bound; it SHALL process reports in waves if the runtime cannot execute them all concurrently.

Required outcomes (why: correctness + traceability):

- **Report ingestion:** each completed review report SHALL be read and its claims SHALL be verified against the current code at the cited anchors.
- **Disposition coverage:** each distinct failure mode SHALL receive a disposition decision (apply/decline/defer) with rationale and verification implications.
- **Deterministic state:** progress and dispositions SHALL be recorded via `mpcr` (applicator notes/status) instead of ad-hoc state.

IF outputs appear shallow OR the change is high-risk THEN the orchestrator SHALL increase depth until the saturation stop condition is met (including by using the depth-increase techniques in Section VIII when useful).

## V) Subagent output contract (Proof Packet)

Subagents SHALL return exactly one Proof Packet and nothing else.

The Proof Packet SHALL:

- be valid Markdown
- begin with the exact header: `## Proof Packet: <subagent name>`
- include the required sections below (in order)

Non-normative guidance: Subagents SHOULD keep Proof Packets concise (a typical packet targets ≤ 5 findings and ≤ 2 pages) unless additional unique failure modes continue to appear (saturation stop condition).
Subagents SHOULD use counterexample-first style: lead with the concrete disproof attempt before conclusions.

### Proof Packet template

## Proof Packet: <subagent name>

- **Assignment:** <what you were asked to do>
- **Scope:** <files/anchors covered + what you did NOT cover>
- **Domains:** <which domains you covered>

### Findings (failed proofs)

- <SEVERITY>: <short title>
  - **Anchor:** <file:line + symbol>
  - **Theorem:** <must-prove claim>
  - **Concrete disproof attempt:** <specific payload / sequence / scenario>
  - **Reproducer:** <minimal input/sequence> (SHALL be included for BLOCKER/MAJOR findings)
  - **Outcome:** <why it fails OR why it holds>
  - **Evidence:** <anchors>
  - **Recommendation:** <specific fix>
  - **How to verify:** <test/command/artifact>

(Repeat for each unique failure mode.)

### Passing proofs (defended theorems)

- <Theorem>
  - **Anchor:** ...
  - **Concrete disproof attempt:** ...
  - **Evidence:** ...
  - **Confidence:** Verified / Supported

### Residual risk

- <Assumed/Unknown claim>
  - **Why it matters:** ...
  - **Verification plan:** ...

### Questions for orchestrator

- ...

## VI) Dispatch content (outcome-focused)

WHEN the orchestrator launches a subagent THEN the orchestrator SHALL provide the assignment objective (WHAT/WHY), scope, and domains.  
IF the subagent is a Child Reviewer THEN the orchestrator SHALL also provide the mpcr session context and the explicit pass-through requirements.  
The following is a **non-normative template** you may use.

You are <Subagent | Child Reviewer> <NAME>.  
Assignment objective: <WHAT you SHALL prove + WHY it matters>  
Scope: <FILES / ANCHORS>  
Domains: <DOMAINS>

MPCR (only if you are delegated as a Child Reviewer):

- session_dir: <DIR>
- target_ref: <REF>
- session_id: <SESSION_ID8>
- reviewer_id: <CHILD_REVIEWER_ID8> (OPTIONAL; omit to let `mpcr reviewer register` generate one)
- parent_id: <PARENT_REVIEWER_ID8>
- Requirement: pass the session context explicitly; set `parent_id` to the orchestrator reviewer id; use the resulting `reviewer_id` (assigned or generated) for all subsequent `mpcr reviewer ...` commands.

Non-normative command sequence (Child Reviewer only; use the assigned ids):

```sh
# If reviewer_id is pre-assigned by the orchestrator:
mpcr reviewer register --target-ref '<REF>' --session-dir <DIR> --session-id <SESSION_ID8> --reviewer-id <CHILD_REVIEWER_ID8> --parent-id <PARENT_REVIEWER_ID8> --print-env
# If reviewer_id is NOT pre-assigned, omit --reviewer-id and use the printed MPCR_REVIEWER_ID afterward:
# mpcr reviewer register --target-ref '<REF>' --session-dir <DIR> --session-id <SESSION_ID8> --parent-id <PARENT_REVIEWER_ID8> --print-env
mpcr reviewer update --session-dir <DIR> --reviewer-id <CHILD_REVIEWER_ID8> --session-id <SESSION_ID8> --status IN_PROGRESS --phase INGESTION
# ... review work ...
mpcr reviewer finalize --session-dir <DIR> --reviewer-id <CHILD_REVIEWER_ID8> --session-id <SESSION_ID8> --verdict <VERDICT> --blocker 0 --major 0 --minor 0 --nit 0 <<'EOF'
## Adversarial Code Review: <Ref>
... full report content ...
EOF
```

Rules:

- If Subagent: you SHALL remain read-only; you SHALL NOT run mpcr; you SHALL NOT edit repo files unless explicitly granted write access by the orchestrator (if granted, you SHALL follow the stated scope and merge strategy).
- If Child Reviewer: you SHALL use `mpcr` ONLY for `mpcr reviewer register/update/note/finalize` and ONLY with your assigned ids; you SHALL pass `--session-dir` and `--session-id` explicitly on every command; you SHALL NOT run `mpcr applicator ...`; you SHALL NOT edit repo files.
- You SHALL anchor every non-trivial claim to code locations.
- You SHALL include at least one concrete counterexample attempt per theorem.

Optional non-normative reminders (include only if relevant):

- Do not run mpcr commands unless explicitly delegated as a Child Reviewer.
- Do not edit repo files unless explicitly granted write access to specific files/anchors.
- Do not invent anchors; use symbol + unique snippet if line numbers are unknown.
- Do not exceed the assigned scope.

Deliverable:

- You SHALL return exactly one artifact:
  - Subagent → one Proof Packet (template in `references/multi-agent.md` section V)
  - Child Reviewer → one UACRP report (template in `references/perform/uacrp.md`)

Output format requirement (Subagents):

- Your response SHALL begin with: `## Proof Packet: <NAME>`

## VIII) Depth increase (triggered; outcome-driven)

WHEN the change is high-risk OR outputs appear shallow THEN the orchestrator SHALL increase depth until the saturation stop condition is met.
This section lists non-normative techniques that can be used to achieve that outcome.

### 1) Counterexample-first reporting

- Subagents and Child Reviewers SHOULD begin each theorem with the concrete disproof attempt and only then state outcome (why: auditability and reduced hand-waving).

### 2) Redundant review on one depth target

IF the change is high-risk THEN there SHALL exist at least two independent analyses of the single highest-risk depth target.
- The orchestrator SHALL surface any disagreement explicitly (do not silently average conclusions).

### 3) Proof-carrying findings

For each BLOCKER/MAJOR finding (Proof Packet or Child Reviewer report):

- there SHALL exist a minimal reproducer (input payload, call sequence, or state-machine trace), AND
- there SHALL exist an explicit “How to verify” plan that would fail if the bug still exists.

### 4) Noise budget (anti-bloat)

- Non-normative guidance: Subagents SHOULD aim for concise findings (commonly ≤ 5) by default.
- IF additional unique failure modes keep appearing THEN subagents SHALL continue until saturation, then stop.
