# Patterns and Anti-Patterns

## Good patterns

1. Two-stage form flow
- Stage 1 (`form1`) collects intent and coarse settings.
- Stage 2 layout is generated by runtime provider logic.
- Stage 2 (`form2`) binds known field definitions to generated placeholders.

2. Provider-dispatch architecture
- Parse `operation` + `provider` first.
- Dispatch to provider-specific layout function.
- Keep each provider focused and testable.

3. Deterministic layout generation
- Stable ordering of dynamic sections and fields.
- No timestamps/random ids in layout text.

4. Stable field names
- Keep provider field keys lowercase and explicit.
- Use `ESPANSO_FORM_FIELD_<name>` consistently across YAML and generator logic.

5. Output correctness
- Generator stdout should be layout text only.
- Error text should go to stderr and non-zero exit.

## Anti-patterns

1. Repo-coupled assumptions
- Hardcoding local absolute paths in docs or scripts.
- Referring to one project's migration history as universal guidance.

2. Status-text payload leakage
- Returning `Copied to clipboard` as expansion output where actual payload is expected.

3. Blocking clipboard writes in hot path
- Waiting synchronously on clipboard ownership commands during text replacement.

4. Inconsistent contract keys
- Mixing ad hoc env names with standardized contract keys in the same flow.

5. Unbounded dynamic fields
- Generating uncontrolled placeholder counts without clear limits or defaults.
